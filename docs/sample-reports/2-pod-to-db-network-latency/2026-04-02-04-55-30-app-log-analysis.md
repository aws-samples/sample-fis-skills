- 恢复后响应时间立即恢复到 2-3ms，说明应用没有连接池预热延迟问题

## 建议

1. **Catalog 错误码修正**
   - **影响:** 数据库超时返回 HTTP 404 使上游无法区分 "数据不存在" 和 "服务故障"
   - **建议:** 修改 `repository.go` 的错误处理，数据库连接/查询错误应返回 HTTP 503 (Service Unavailable)

2. **UI 服务增加断路器**
   - **影响:** catalog 故障时 UI 2546 次异常导致用户体验完全中断
   - **建议:** 使用 Resilience4j CircuitBreaker 包装 catalog 客户端调用，故障时返回缓存的商品数据或友好的降级页面

3. **增加连接超时的合理性**
   - **影响:** catalog 的 MySQL 连接超时 5s，在 20s 延迟下每次连接必定失败
   - **建议:** 结合重试策略（如指数退避）和连接池保活机制，减少对新连接的依赖

4. **添加可观测性**
   - **影响:** 缺少结构化日志和 Metrics，故障排查需要大量日志搜索
   - **建议:** 为 catalog 添加 Prometheus 指标（数据库连接成功率、查询延迟 P99）和结构化 JSON 日志
